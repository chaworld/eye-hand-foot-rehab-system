# 基本規則

本文件定義本專案（`eye_hand_foot`）的最小協作規範，所有參與者（開發者與 AI Agent）皆應遵守。

## 1. 開發原則
- 先理解需求，再修改程式。
- 優先做最小可行變更，避免大幅重構無關程式。
- 不破壞既有功能：新功能加入後需能維持現有流程可用。
- 以可讀性優先：命名清楚、函式單一職責、避免過深巢狀。

## 2. 檔案與模組規範
- 主要入口為 `app.py`，請避免在其他模組中重複建立啟動流程。
- 介面相關邏輯集中於 `gui.py`、`gui_module.py`、`foot_gui.py`。
- 偵測相關邏輯集中於 `eye_tracker.py`、`hand_detector.py`、`foot_detector.py`。
- 音訊與語音功能集中於 `voice_assistant.py` 與 `audio/`。
- `release_code/` 視為發佈版本，不直接當作日常開發主目錄。

## 3. 程式風格
- 使用 Python 3，維持與既有程式一致的風格。
- 縮排一律 4 個空白，不使用 Tab。
- 新增函式需加上簡短註解或 docstring，說明用途與輸入輸出。
- 避免硬編碼路徑，優先使用相對路徑與 `os.path`/`pathlib`。

## 4. 相依套件與環境
- 新增套件時必須同步更新 `requirements.txt`。
- 不提交個人環境檔案（例如本機暫存、IDE 私有設定）。
- 音訊資源若有新增，請更新 `README.md` 的音檔說明。

## 5. 測試與驗證
- 每次修改後，至少執行：`python app.py`，確認程式可啟動。
- 若修改偵測模組，需確認對應功能在 GUI 中可觸發且不報錯。
- 若修改語音模組，需確認 TTS/音效可正常播放或有合理 fallback。
- 發生例外時，優先修正根因，不用 `try/except` 直接吞錯。

## 6. Git 與提交規範
- 每次提交只做一件事，訊息清楚描述目的。
- 提交訊息建議格式：`type(scope): summary`
- 常用 type：`feat`、`fix`、`refactor`、`docs`、`chore`。
- 不得覆蓋或還原非本次任務的既有變更。

## 7. 文件同步
- 需求、流程或操作方式有變更時，必須同步更新：
	- `README.md`
	- `docs/代辦.md`（如影響待辦）
	- `docs/部屬步驟.md`（如影響部署）

## 8. AI Agent 額外規則
- 修改前先閱讀相關檔案，避免盲改。
- 優先使用小範圍 patch，避免無關格式化。
- 若發現與任務無關的異常變更，先停止並回報。
- 回覆需明確列出：修改檔案、修改原因、驗證結果。

---

若規則與實際開發需求衝突，以「不破壞功能、可維護、可驗證」三原則做最終判斷。
